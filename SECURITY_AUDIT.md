# Security Audit

Scope: this repository is a React/Vite real-estate website with a small Express API for lead intake, valuation, and booking. Listing admin controls are still demo-only client state; a production admin should require authenticated server-side authorization.

## 1. Authentication & Authorization

No user login, JWT, password storage, or session code exists in this version. The Express API accepts public lead, valuation, and booking requests only. The listing admin UI is intentionally demo state and does not write to the server.

Implemented control:

```jsx
// server/index.js
app.post("/api/leads", (req, res) => {
  const lead = assertLead(req.body);
  leads.push({ ...lead, id: randomUUID(), createdAt: new Date().toISOString() });
  res.status(201).json({ ok: true, message: "Lead received." });
});
```

Required backend pattern if accounts are added:

```js
app.get("/api/me", requireAuth, async (req, res) => {
  const user = await users.findById(req.user.id).select("id name email role");
  res.json({ user });
});
```

Severity: Low for current public API. High if persistent admin routes are later added without server-side authorization middleware.

## 2. Injection & Input Validation

Finding: public lead, valuation, and booking forms accept user-controlled text. React escapes values by default, and the backend repeats sanitization and validation before storing in memory.

Affected code:

```jsx
<input name={name} value={value} onChange={onChange} maxLength={MAX_FIELD_LENGTH} />
```

Proof-of-concept attack string:

```txt
"><script>alert(1)</script>
```

Secure implementation:

```jsx
function sanitizeInput(value, max = MAX_FIELD_LENGTH) {
  return value
    .replace(/[<>{}`]/g, "")
    .replace(/\s+/g, " ")
    .trimStart()
    .slice(0, max);
}
```

Controls implemented:
- React rendering, no `dangerouslySetInnerHTML`.
- Length limits on all user-facing inputs.
- Email, phone, ZIP, message, consent, and honeypot validation.
- Express JSON body limit of `24kb`.
- No SQL, NoSQL, command execution, file paths, URL fetching, or SSRF sink exists.
- Backend sanitizes `<>{}\`` and trims input before storing.

Severity: Low currently. Medium if this later connects to a database without parameterized queries or schema validation.

## 3. API & Data Exposure

The API returns only `ok`, `message`, or valuation range fields. It does not return stored lead or booking records.

Implemented controls:

```json
// vercel.json
{
  "key": "X-Frame-Options",
  "value": "DENY"
}
```

```txt
# public/_headers
Content-Security-Policy: default-src 'self'; script-src 'self'; ...
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
X-Content-Type-Options: nosniff
```

Risk to address if a backend is added:
- Add persistent storage with encryption-at-rest if leads/bookings are saved beyond process memory.
- Do not use wildcard CORS on authenticated routes.
- Return generic error messages to clients.
- Paginate any listing or inquiry APIs.
- Never return passwords, password hashes, reset tokens, auth tokens, or internal notes.

Severity: Low currently. Medium to Critical depending on future persistent API behavior.

## 4. Dependency & Configuration

Dependencies are minimal and pinned through `package.json`. There are no committed `.env` files, no eval-like dynamic execution, and no production server process requiring elevated OS privileges. `helmet`, `cors`, and `express-rate-limit` protect the API layer.

Implemented controls:

```json
"scripts": {
  "build": "vite build",
  "lint": "eslint ."
}
```

Commands to run during maintenance:

```bash
npm audit
npm outdated
npm update
```

Priority remediation if vulnerabilities appear:
1. Upgrade affected packages with `npm update`.
2. Replace deprecated or malicious packages.
3. Keep production builds generated by `npm run build`.
4. Store secrets only in hosting-provider environment variables, never in `src/`.

Verification: `npm audit --audit-level=moderate` completed with zero vulnerabilities after dependency installation.

Severity: Low currently.

## 5. Business Logic & State Management

The app has no payments, pricing mutations, database writes, file uploads, or server-trusted IDs. The only stateful workflow is the public inquiry form.

Implemented controls:

```jsx
if (now - lastSubmitAt < 8000) {
  setStatus("Please wait a moment before submitting again.");
  return;
}
```

Risk to address if dynamic features are added:
- Recalculate prices and commissions server-side.
- Add idempotency keys to payment or booking operations.
- Validate uploaded file MIME type, size, and content server-side.
- Prevent IDOR by scoping records to `req.user.id`.
- Add audit logs for login, deletion, permission changes, and valuation exports.

Severity: Low currently. High if future client-side values are trusted by a backend.
